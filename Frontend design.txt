This skill guides creation of distinctive, production-grade frontend interfaces that avoid generic "AI slop" aesthetics. Implement real working code with exceptional attention to aesthetic details and creative choices.

The user provides frontend requirements: a component, page, application, or interface to build. They may include context about the purpose, audience, or technical constraints.

## Design Thinking

Before coding, understand the context and commit to a BOLD aesthetic direction:
- **Purpose**: What problem does this interface solve? Who uses it?
- **Tone**: Pick an extreme: brutally minimal, maximalist chaos, retro-futuristic, organic/natural, luxury/refined, playful/toy-like, editorial/magazine, brutalist/raw, art deco/geometric, soft/pastel, industrial/utilitarian, etc. There are so many flavors to choose from. Use these for inspiration but design one that is true to the aesthetic direction.
- **Constraints**: Technical requirements (framework, performance, accessibility).
- **Differentiation**: What makes this UNFORGETTABLE? What's the one thing someone will remember?

**CRITICAL**: Choose a clear conceptual direction and execute it with precision. Bold maximalism and refined minimalism both work - the key is intentionality, not intensity.

Then implement working code (HTML/CSS/JS, React, Vue, etc.) that is:
- Production-grade and functional
- Visually striking and memorable
- Cohesive with a clear aesthetic point-of-view
- Meticulously refined in every detail

## Frontend Aesthetics Guidelines

Focus on:
- **Typography**: Choose fonts that are beautiful, unique, and interesting. Avoid generic fonts like Arial and Inter; opt instead for distinctive choices that elevate the frontend's aesthetics; unexpected, characterful font choices. Pair a distinctive display font with a refined body font.
- **Color & Theme**: Commit to a cohesive aesthetic. Use CSS variables for consistency. Dominant colors with sharp accents outperform timid, evenly-distributed palettes.
- **Motion**: Use animations for effects and micro-interactions. Prioritize CSS-only solutions for HTML. Use Motion library for React when available. Focus on high-impact moments: one well-orchestrated page load with staggered reveals (animation-delay) creates more delight than scattered micro-interactions. Use scroll-triggering and hover states that surprise.
- **Spatial Composition**: Unexpected layouts. Asymmetry. Overlap. Diagonal flow. Grid-breaking elements. Generous negative space OR controlled density.
- **Backgrounds & Visual Details**: Create atmosphere and depth rather than defaulting to solid colors. Add contextual effects and textures that match the overall aesthetic. Apply creative forms like gradient meshes, noise textures, geometric patterns, layered transparencies, dramatic shadows, decorative borders, custom cursors, and grain overlays.

NEVER use generic AI-generated aesthetics like overused font families (Inter, Roboto, Arial, system fonts), cliched color schemes (particularly purple gradients on white backgrounds), predictable layouts and component patterns, and cookie-cutter design that lacks context-specific character.

Interpret creatively and make unexpected choices that feel genuinely designed for the context. No design should be the same. Vary between light and dark themes, different fonts, different aesthetics. NEVER converge on common choices (Space Grotesk, for example) across generations.

**IMPORTANT**: Match implementation complexity to the aesthetic vision. Maximalist designs need elaborate code with extensive animations and effects. Minimalist or refined designs need restraint, precision, and careful attention to spacing, typography, and subtle details. Elegance comes from executing the vision well.

Global Shortcuts

- Arrow keys → navigate lists
- Tab → logical focus order

Component-Level Shortcuts
Examples:

Accessibility Requirements

- Logical tab order
- No keyboard traps
- ARIA where appropriate

## Appendix: Anti-Card and Anti-Gradient Enforcement + Strengthened Anti-Slop Measures

This appendix is mandatory and must be read and applied in full before any frontend implementation begins. It extends and overrides any conflicting guidance in the main document where necessary. Its purpose is to eliminate the two most persistent and destructive failure modes observed in LLM-generated interfaces: excessive card-based layouts and decorative gradients. These patterns produce the exact generic, floating, modular “AI slop” that the core document was written to prevent.

All rules below are non-negotiable and form part of the Design System Enforcement and Anti-AI Slop Safeguard sections.

### 1. Cardless Design Mandate (Primary Visual Rule)
Default to a completely cardless interface. Cards (defined as any self-contained, bordered, shadowed, or background-isolated rectangular container that visually floats or nests) are banned as the default structural element.

- Never use cards as the primary visual motif.
- Never nest cards inside other cards.
- Never wrap every section, panel, or content block in a card-like container.
- Replace cards with stronger architectural primitives: monolithic slabs, full-width sections, integrated panels, fieldsets, or naked content areas separated only by spatial composition (whitespace, subtle dividers, background colour blocks, or deliberate overlap).
- Containment must be achieved through layout grid behaviour, generous negative space, asymmetry, or intentional density contrast — never through repeated bordered/shadowed boxes.
- Edge-case allowance: cards may be used only when semantically required for grouping a small set of highly interactive elements (e.g., a tightly coupled form group or data tile set) and only if they are architecturally integrated into the larger layout rather than floating as islands. In such cases they must use the exact border-radius and shadow tokens defined in the design system and must not introduce new visual hierarchy.

This rule directly addresses the user’s explicit aversion to “cards inside cards” and the entire page becoming a card. It forces spatial composition, asymmetry, overlap, and monolithic forms as required by the main document’s Spatial Composition and Frontend Taste Calibration guidelines.

### 2. Gradient Ban
All decorative or surface-level gradients are strictly prohibited.

- Surfaces (backgrounds, panels, cards if exceptionally allowed, buttons, headers) must use solid colours only, or non-gradient techniques: grain/noise overlays, subtle fabric-like textures (via CSS), layered transparencies, or architectural box-shadows.
- Linear, radial, or conic gradients are banned except for tiny accent elements no larger than 1–2 px (e.g., a progress-bar fill or a 1 px hover underline).
- Explicitly forbidden: purple gradients, peach gradients, warm beige gradients, “glassmorphic” or “neubrutalist” gradient overlays, and any gradient used to simulate depth on large surfaces.
- Colour system must remain flat and solid unless the aesthetic direction explicitly justifies a non-gradient texture (e.g., paper grain for an editorial aesthetic or lacquered wood for an instrument-related tool).

This ban prevents the clichéd, soft, wellness-app aesthetics that frequently appear in onboarding flows and productivity interfaces.

### 3. Mandatory Negative Example Rejection Step
Immediately after choosing the aesthetic direction (but before defining design tokens), the agent must perform and visibly output a “Negative Example Rejection” analysis using the following exact format:

**Negative Example Rejection**  
The generic failure mode this design explicitly rejects is: [one-sentence description of the exact pattern the model was about to produce, e.g., “three-panel floating card layout with soft beige backgrounds, generous whitespace, and nested rounded rectangles typical of generic productivity onboarding”].  
This interface will instead achieve memorability through: [specific non-card, non-gradient technique tied to the signature move].

This step forces the model to name the precise monstrosity it is avoiding before proceeding.

### 4. Strengthened Signature Move Requirement
Every design must have one clear, memorable signature move. This move must:
- Directly break the card paradigm (e.g., grid-breaking overlap, diagonal flow, monolithic slab layout, custom spatial rhythm, or unexpected layering).
- Be derived from the domain context (see section 5 below).
- Be explicitly justified in one sentence before design tokens are defined: “The signature move is [description] and it ensures the interface feels purpose-built rather than modular.”

### 5. Enhanced Domain-Derived Taste Calibration
Before selecting the aesthetic direction, the agent must first derive visual language from real-world references specific to the product domain. For any music/creative-practice tool (or similar tactile domain), list 3–5 concrete physical or cultural references (e.g., lacquered piano keys, annotated music stands, wooden practice journals, metronome ticks, heavy manuscript paper) and explicitly state how each informs colour, texture, typography, or spatial decisions.  

This calibration must be output visibly and must shape the choice of fonts, surfaces, and motion. It prevents defaulting to soft-fabric productivity aesthetics and ensures the design feels genuinely context-specific.

### 6. Updated Preflight Check (Now Visible and Mandatory)
Before outputting any code, the agent must run and display this exact preflight checklist with yes/no answers and one-sentence justifications:

**Preflight Check**  
- Aesthetic thesis is clear and extreme: [yes/no]  
- Information density model is explicit and applied: [yes/no]  
- Design system tokens are defined and used globally: [yes/no]  
- Cardless design mandate is satisfied (no floating or nested cards): [yes/no]  
- Gradient ban is satisfied (no decorative gradients on surfaces): [yes/no]  
- Signature move is present and non-card-based: [yes/no]  
- Keyboard usability and accessibility are engineered: [yes/no]  
- Layout will scale without collapse: [yes/no]  
- Would a skilled human designer respect this as intentional work: [yes/no]  

If any answer is no, the design must be revised before code is generated.

### 8. Final Enforcement Language
This appendix has equal weight to the main document. Any generated interface that violates the cardless mandate or gradient ban is considered failed output, regardless of how closely it follows other rules. The agent must treat these constraints as absolute filters before producing code. The result must feel like a cohesive, purpose-built architectural composition — never a collection of floating modular components.

All other rules from the main document (British English, emoji ban, Tailwind preference, keyboard-first model, accessibility, real data, motion discipline, etc.) remain in force and are reinforced by this appendix.
